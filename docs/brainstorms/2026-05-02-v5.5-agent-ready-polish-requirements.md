## Change 1 — Nuonuo regex fix

### Problem

`scripts/invoice_helpers.py:437-446` picks up the wrong URL from Nuonuo invoice
emails. The regex `[\w\-=]+` doesn't include `/`, so
`https://nnfp.jss.com.cn/allow/service/getEwmImg.do?content=…` matches as
`https://nnfp.jss.com.cn/allow` (the QR image URL). The real short link
appears later in the email body but `urls[0]` wins. Downstream
`resolve_nuonuo_short_url` GETs `/allow`, gets HTTP 200 instead of 302, returns
None → `"failed to resolve nuonuo short link"`.

Current filter catches `/scan-invoice/` but not `/allow`, `/invoice`, `/scan`,
`/sapi`.

### Design

Extend the exclusion filter (approach A from the Feishu doc). Reject
approach B (minimum code length) — it stacks false-negative risk with no
additional safety. Losing a real invoice (silent data loss) is worse than
letting an API URL leak through (loud `failed to resolve` error that probe
can diagnose).

```python
def extract_nuonuo_short_url(body):
    """Find the 诺诺网 short link in email body. Returns marker for
    two-step resolution.

    Excludes API/utility paths (/allow /invoice /scan /sapi /scan-invoice)
    that share the nnfp.jss.com.cn host but are not redirectable short links.
    """
    urls = re.findall(r'https?://nnfp\.jss\.com\.cn/[\w\-=]+', body)
    EXCLUDED = ('/allow', '/invoice', '/scan', '/sapi', '/scan-invoice')
    urls = [u for u in urls if not any(p + '/' in u or u.endswith(p) for p in EXCLUDED)]
    if urls:
        return f"NUONUO_SHORT:{urls[0]}"
    return None
```

### Out of scope

- `resolve_nuonuo_short_url` (the 302-follower) — works correctly given a real
  short link, per the Feishu doc's Step 2-4 validation.
- `of1.cn/{code}` alternate short-link domain mentioned in the Feishu
  "边界情况 & 风险" callout — separate platform addition, defer.
- The `NUONUO_SHORT:` marker contract with `download-invoices.py:426` is
  unchanged.

### Tests

In the existing platform-extraction test module:

1. Golden bug repro: body contains both the `/allow/service/getEwmImg.do` QR URL
   and the real short link → extractor returns the real short link.
2. Allow-only body: body contains only the QR URL → returns None (doesn't
   misidentify `/allow` as a short link).
3. `/scan-invoice/` filter still works (regression guard, add if not present).

## Change 2 — Agent auto-probe loop + `--postprocess-only` + unknown-platform surfacing

### Problem

Today, when the script can't resolve a short link for a known platform, it
logs `failed to resolve … short link` and moves on. The Agent reads
`missing.json`, sees no actionable `search_suggestion`, and either stops or
runs a no-op supplemental. `probe-platform.py` exists as a diagnostic tool
but only runs when a developer invokes it manually.

Second, if probe *also* can't find a PDF URL (unknown / unsupported platform),
the Agent has no channel to surface that to the user — it just silently
accepts the partial result.

### Design

**SKILL.md-only, plus two small code-side enablers.**

- The probe workflow stays in the Loop Playbook (Agent-driven, not baked
  into `download-invoices.py`) — per the rationale in the Feishu doc's
  "为什么不在脚本里做" section: probe is non-deterministic, Agent can judge
  plausibility, script stays deterministic.
- Two enablers: (a) `--postprocess-only` flag so Agent can re-run matching
  after curling a rescued PDF into `pdfs/`, and (b) a helper script
  `scripts/record-unknown-platform.py` so Agent can correctly append an
  `unknown_platform` entry to `missing.json` and recompute `convergence_hash`
  without hand-editing JSON.

#### SKILL.md — new "Failed-link auto-probe" subsection (after Loop 决策)

```
### Failed-link auto-probe (exit 5, before run_supplemental)

When run.log contains "failed to resolve … short link" entries, iterate each:

1. Extract the original URL from the run.log entry.
2. Run: python3 scripts/probe-platform.py "<url>"
3. If probe outputs a PDF URL:
     a. curl -sL --max-time 60 -H "User-Agent: Mozilla/5.0" \
          -o <out>/pdfs/probe_<msgid>.pdf "<pdf_url>"
     b. Validate the %PDF header (first 4 bytes).
     c. On success: re-run postprocess only:
          python3 scripts/download-invoices.py --postprocess-only --output <out>
     d. If the rescued item now appears in matching, done.
4. If probe cannot find a PDF URL (unknown platform / broken chain):
     # Source: run.log carries the URL + msgid. Email subject/from come
     # from <out>/step3_classified.json (classification stage output),
     # which is written for every run.
     python3 scripts/record-unknown-platform.py \
        --output <out> \
        --url "<url>" \
        --email-subject "<subject>" \
        --email-from "<from>" \
        --probe-suggestion "<probe stdout 下一步建议>"
     This appends an item with type=unknown_platform to missing.json.items[],
     flips status to user_action_required, and recomputes convergence_hash.

After probing all failed links:
- If all resolved → re-read missing.json, continue Loop decisions.
- If any remained unresolved → missing.json.status is user_action_required.
  The loop ends. In the final chat summary, list:
    - original email subject + sender
    - platform host (from probe)
    - probe's "下一步建议" output
    - pointer to references/platforms.md for the 5-step new-platform playbook
```

#### SKILL.md — Loop Playbook switch case update

Existing:
```
case "user_action_required":  → DONE（有 extraction_failed，提示人工检查）
```

Updated:
```
case "user_action_required":
    → DONE。items 中可能包含:
      - type=extraction_failed — LLM 读不出的 PDF，人工核查文件是否损坏/是否发票
      - type=unknown_platform — probe 也无法识别的新平台，按清单提示用户，
        建议参考 references/platforms.md 的 5-step playbook 评估是否扩展
```

#### SKILL.md — Exit Codes table update

`5 | 部分成功` row REMEDIATION note appended:
`→ 先跑 auto-probe（见 Loop Playbook），再决定 run_supplemental`.

#### `--postprocess-only` flag

New CLI flag in `download-invoices.py`. Semantics: skip Step 1–5 (Gmail
search + download). Run Step 6–10 only, reading the already-downloaded PDFs
from `<output>/pdfs/` and the existing `step4_downloaded.json` if present.

- Leverages the existing OCR cache — already-analyzed PDFs are 0-cost on
  re-run.
- Newly curled PDFs (probe rescues) have no cache entry, trigger fresh OCR.
- Updates the three deliverables (`下载报告.md`, `发票汇总.csv`,
  `missing.json`) and re-zips with the same allowlist.
- Exit codes semantics unchanged.
- Empty / non-existent `<output>/pdfs/` → writes empty report, exits 5 with
  a REMEDIATION explaining the empty state. Does not crash.

#### `scripts/record-unknown-platform.py` helper

Small single-purpose script. Accepts `--output`, `--url`, `--email-subject`,
`--email-from`, `--probe-suggestion` and:

1. Reads existing `<output>/missing.json`.
2. Appends to `items[]`:
   ```json
   {
     "type": "unknown_platform",
     "needed_for": "<url>",
     "email_subject": "...",
     "email_from": "...",
     "original_url": "...",
     "probe_suggestion": "...",
     "hint": "未知平台，probe 无法自动恢复。按 references/platforms.md 5-step playbook 评估是否扩展支持",
     "search_suggestion": null
   }
   ```
3. Sets `status = "user_action_required"`, `recommended_next_action = "ask_user"`.
4. Recomputes `convergence_hash` (per the existing formula in `postprocess.py`).
5. Atomic write via `.tmp → os.replace`.

Refuses to run if `missing.json` doesn't exist or has unrecognized schema_version.

### Schema notes

- `unknown_platform` is a new `type` value. v1.0 schema allows — the `type`
  field was never a closed enum in the original spec, and the Loop Playbook
  already treats unrecognized types by routing through the top-level `status`.
- `status=user_action_required` and `recommended_next_action=ask_user` are
  existing enum values; no schema extension needed.

### Out of scope

- Agent autonomously editing `invoice_helpers.py` to support a new platform.
  That remains a human-reviewed `references/platforms.md` 5-step playbook.
  Agent only detects and surfaces; doesn't fix code.
- Re-running Step 1-5 from `--postprocess-only`. The whole point is to
  *not* re-search Gmail.

### Tests

In `tests/test_agent_contract.py`:

1. `--postprocess-only` on a directory with 0 PDFs writes a coherent empty
   report and exits 5 with the expected REMEDIATION line.
2. `--postprocess-only` on a directory with existing `pdfs/` reuses the OCR
   cache, recomputes matching, and updates the three deliverables.
3. `record-unknown-platform.py` appends an item with correct shape, updates
   top-level `status`/`recommended_next_action`, and recomputes
   `convergence_hash`.
4. `record-unknown-platform.py` refuses to run on missing / malformed
   `missing.json` (exits non-zero with REMEDIATION).

## Change 3 — P3 OCR dates in report; folio filename uses `departureDate`

### Problem

P3 (date-only fallback) matches look wrong on inspection. Example from the
Feishu doc:

| 退房日 | 销售方 | 匹配方式 | 水单 | 发票 |
|---|---|---|---|---|
| 2025-05-08 | 苏州万豪 | P3 (仅日期)⚠️ | 20250607_苏州万豪_水单.pdf | 20250508_苏州万豪_发票.pdf |

The folio filename starts with `20250607` (email `internalDate` — flight
invoice platforms sometimes send folios a month after checkout). The actual
OCR `departureDate` is 2025-05-08, which the matcher correctly used. Reviewer
sees `0607 ≠ 0508` and suspects mis-match.

### Design

Two independent sub-changes:

#### 3A — P3 report rows show OCR folio arrival/departure dates

Insert one column for P3 rows only (P1/P2 unchanged — filename dates
already agree with OCR dates for those tiers, so adding a column would be
noise).

| 销售方 | 匹配方式 | 入住日 / 退房日 (OCR) | 水单 | 发票 |
|---|---|---|---|---|
| 苏州万豪 | P3 (仅日期)⚠️ | 2025-05-07 / 2025-05-08 | 20250607_苏州万豪_水单.pdf | 20250508_苏州万豪_发票.pdf |

Rationale for showing the folio's arrival/departure span (not also the
invoice's `transactionDate`): the invoice filename already encodes its own
date. The genuinely new information is the folio's check-in/check-out window
— it explains why the P3 fallback is legitimate ("one-night stay, invoice
issued on checkout day").

Edge cases:
- Folio with only `departureDate` → show as `? / 2025-05-08`.
- Folio with neither → render today's filename-date behavior and leave a
  `?` marker so reviewers know to open the PDF.

Data plumbing: `do_all_matching` in `postprocess.py` augments each P3 match
record with `folio_arrival_date` and `folio_departure_date` fields read from
the folio's OCR dict. The report writer in `download-invoices.py` renders
these when building the P3 table. No changes to `missing.json` or CSV.

#### 3B — Folio filename uses OCR `departureDate`

`scripts/postprocess.py::rename_by_ocr`: for `HOTEL_FOLIO` category, prefer
OCR `departureDate` over email `internalDate` when constructing the
filename's date prefix.

```python
if category == "HOTEL_FOLIO" and _to_yyyymmdd(ocr.get("departureDate")):
    date_part = _to_yyyymmdd(ocr["departureDate"])
else:
    date_part = internal_yyyymmdd
```

- All other categories unchanged.
- Existing collision-suffix logic (`_1`, `_2`) handles multi-folio same-day
  same-hotel legitimately.

### Out of scope (explicitly rejected)

- Amount-diff column on P3 rows (rejected during brainstorming).
- P1/P2 OCR date columns (noise, dates already align).
- CSV date-column changes (already uses OCR dates).
- Rewriting existing files when migrating from v5.4. User re-running a batch
  in a new output dir gets new names; running `--postprocess-only` on an
  existing dir will produce new names alongside old ones (one-shot manual
  cleanup acceptable).

### Tests

In `tests/test_postprocess.py`:

1. P3 match record has `folio_arrival_date` + `folio_departure_date`
   populated from folio OCR (not from filename).
2. Folio with `internalDate=2025-06-07` + OCR `departureDate=2025-05-08`
   → filename starts with `20250508_`.
3. Folio with `departureDate=null` → filename falls back to `internalDate`
   YYYYMMDD (current behavior preserved).
4. Non-folio invoice with some `departureDate` noise → filename unaffected,
   still uses v5.3 logic.

### Risk

Files named with v5.4 pattern (`20250607_…`) will be absent from v5.5 runs
on the same emails — replaced by `20250508_…`. Users organizing by filename
will see reordering. Called out in CHANGELOG.

## Change 4 — Cross-quarter `out_of_range_items[]`

### Problem

Hotel folios / invoices frequently arrive in email weeks after the actual
stay or billing date. User runs `--start 2026/04/01 --end 2026/07/01` for
Q2. Gmail returns Q2-sent emails that describe Q1 business dates
(`departureDate=2026-03-18` in an email sent 2026-04-10). The folio sits
unmatched because its invoice was sent in Q1's email window and never
downloaded. `missing.json` suggests `run_supplemental`. Agent expands the
search window to March. Result: 6 more Q1 folios drag into the Q2
reimbursement package, creating more "missing" pairs — all Q1, all outside
scope.

### Design

**Filter in `write_missing_json` (`postprocess.py`) before retry
recommendations are computed.** The OCR business date is already attached
to each candidate item at that point via `_record.ocr`.

Business date lookup table (item `type` describes what's *missing* — the
business date comes from what we *have*):

| `type` (missing) | We have | Business date field |
|---|---|---|
| `hotel_folio` | invoice | `transactionDate` |
| `hotel_invoice` | folio | `departureDate` (or `checkOutDate`) |
| `ridehailing_receipt` | invoice | `transactionDate` |
| `ridehailing_invoice` | receipt/itinerary | `transactionDate` |
| `extraction_failed` | unreadable PDF | N/A — never filtered |
| `unknown_platform` (change #2) | N/A | N/A — never filtered |

Filter logic (inclusive start, exclusive end — matches Gmail `before:`
convention):

```python
# OCR business dates arrive as YYYY-MM-DD (canonical OCR format).
# CLI --start / --end arrive as YYYY/MM/DD (Gmail search format).
# Normalize both to date objects.
from core.validation import _parse_ocr_date   # strict YYYY-MM-DD parser

def _parse_cli_ymd(s):
    # YYYY/MM/DD → YYYY-MM-DD → date via existing helper.
    return _parse_ocr_date(s.replace("/", "-")) if s else None

def _is_out_of_range(business_date, run_start, run_end):
    d = _parse_ocr_date(business_date) if business_date else None
    s = _parse_cli_ymd(run_start)
    e = _parse_cli_ymd(run_end)
    if d is None or s is None or e is None:
        return False   # default-in if we can't parse; don't filter by guess
    return d < s or d >= e
```

`_parse_cli_ymd` is a new local helper in `postprocess.py`. It's small
enough not to need its own file but deserves a direct test.

### Output shape

Top-level `out_of_range_items[]` array added to `missing.json`. Same item
shape as `items[]` plus two extra fields:

```json
{
  "out_of_range_items": [
    {
      "type": "hotel_invoice",
      "needed_for": "20250318_杭州万豪_水单.pdf",
      "expected_date": "2025-03-18",
      "expected_merchant": "杭州万豪",
      "business_date": "2025-03-18",
      "reason": "business_date_out_of_range",
      "hint": "水单退房日 2025-03-18 早于本批次 --start 2026/04/01",
      "search_suggestion": null
    }
  ]
}
```

- `status` / `recommended_next_action` are computed from `items[]` only.
- `convergence_hash` is computed from `items[]` only (existing formula,
  existing behavior).
- Empty `items[]` + non-empty `out_of_range_items[]` → `status=converged`,
  `recommended_next_action=stop`.
- `schema_version` stays `"1.0"` — new top-level fields are additive and
  don't break v1.0 consumers.

### Report display

`下载报告.md` gets a new subsection when `out_of_range_items[]` is non-empty:

```
## ℹ️ 跨季度边界项（无需补搜）

以下 N 项的业务日期不在本批次时间范围（YYYY-MM-DD ~ YYYY-MM-DD）内，
已跳过自动补搜。如需一并报销，请单独跑对应季度的批次。

- 20250318_杭州万豪_水单.pdf（退房日 2025-03-18）
- 20250304_杭州万豪_发票.pdf（开票日 2025-03-04）
```

No match-suggestion column — these are explicitly out of scope for this
batch.

### SKILL.md schema documentation

The `missing.json schema v1.0` JSON block in the Loop Playbook section gets
the new `out_of_range_items` array appended with a concise comment.

### Out of scope

- Deleting the out-of-range PDFs from `pdfs/`. They stay — user may want
  to ship them separately.
- Auto-suggesting the correct quarter to re-run. Probably a future UX
  improvement, not v5.5.

### Tests

In `tests/test_postprocess.py`:

1. Unmatched `hotel_invoice` with folio `departureDate` before `run_start`
   → lands in `out_of_range_items`, not `items`. Status `converged` if
   items is empty.
2. In-range folio unchanged: status `needs_retry` (regression guard).
3. Mixed batch: one out-of-range + one in-range → items has in-range,
   out_of_range_items has the other, status `needs_retry`.
4. Missing business_date stays in `items[]` (default-in behavior).
5. `extraction_failed` never routed to `out_of_range_items`.
6. Boundary: `business_date == run_start` stays in-range;
   `business_date == run_end` is out-of-range.
7. `_parse_cli_ymd` round-trips `"2026/04/01"` → `date(2026, 4, 1)`;
   returns None on `""` / `"2026-04"` / `"abc"`.

In `tests/test_agent_contract.py`:

8. Schema contract: parse a synthetic `missing.json` with populated
   `out_of_range_items` — all v1.0 required top-level fields still present,
   `schema_version == "1.0"`.

## Change 5 — OCR concurrency default 2→5 + env var

### Problem

`scripts/postprocess.py:158` — `max_workers: int = 2` — was chosen for
Anthropic tier-1 rate limits. Bedrock (the default provider) has much
higher concurrent-call budgets; 2 throttles Bedrock runs unnecessarily.

Docstring already promises `LLM_OCR_CONCURRENCY` env var support but
nothing reads it.

### Design

Default bumped to 5. Env var honored via a sentinel-None pattern so
explicit callers still override both.

```python
def analyze_pdf_batch(
    records: List[Dict[str, Any]],
    *,
    max_workers: Optional[int] = None,
    use_llm: bool = True,
    logger: Optional[Any] = None,
) -> Dict[str, Dict[str, Any]]:
    if max_workers is None:
        env = os.environ.get("LLM_OCR_CONCURRENCY", "").strip()
        if env:
            try:
                parsed = int(env)
                if parsed < 1:
                    raise ValueError(f"must be >= 1, got {parsed}")
                max_workers = parsed
            except ValueError as e:
                raise LLMConfigError(
                    f"invalid LLM_OCR_CONCURRENCY={env!r}: {e}"
                ) from None
        else:
            max_workers = 5
    ...
```

Precedence: explicit kwarg > env var > default (5).

Invalid env var → `LLMConfigError` (exit code 3). No silent fallback —
matches the CLAUDE.md anti-pattern rule.

### Doctor check

`scripts/doctor.py` gets a new per-env-var check:

- `LLM_OCR_CONCURRENCY` unset → `ok: "using default=5"`.
- Valid int 1-20 → `ok: "LLM_OCR_CONCURRENCY=N"`.
- Valid int >20 → `warn: "unusually high; most providers throttle above 10"`.
- Non-integer or <1 → `fail: ...`.

### SKILL.md

New one-paragraph entry under "LLM Provider":

```
### LLM OCR 并发控制

- 默认 5 并发（Bedrock 默认配额 + 大多数 OpenAI/Anthropic tier-2+ 足够）
- Anthropic tier-1：`export LLM_OCR_CONCURRENCY=2`
- 自建代理/低带宽：按需调低
```

### Tests

In `tests/test_postprocess.py`:

1. `LLM_OCR_CONCURRENCY=3` + no explicit `max_workers` → pool built with 3.
2. Explicit `max_workers=4` beats `LLM_OCR_CONCURRENCY=10` → pool built with 4.
3. `LLM_OCR_CONCURRENCY=abc` → `LLMConfigError` raised.
4. `LLM_OCR_CONCURRENCY=-1` → `LLMConfigError` raised.
5. No env var set → pool built with 5.

In doctor test class:

6. `_check_ocr_concurrency` yields `fail` for `"abc"`, `warn` for `"50"`,
   `ok` for `"3"` and unset.

## Change 6 — Sonnet 4.6 Bedrock default + prompt rules + `applicationDate`

Three sub-changes, all in `scripts/core/`.

### 6A — Bedrock default model: Opus 4.7 → Sonnet 4.6

`scripts/core/llm_client.py:295-296`:

```python
self.model_id = model_id or os.environ.get(
    "BEDROCK_MODEL_ID", "global.anthropic.claude-sonnet-4-6"  # was opus-4-7
)
```

Rationale (per Feishu doc's Q2 95-PDF head-to-head): Sonnet is 42% faster,
~5× cheaper, zero amount-field drift vs Opus. The 8 date inconsistencies
observed are prompt-ambiguity issues addressed in 6B, not model capability.

**Plan-phase action:** verify the exact Bedrock model ID string against
AWS Bedrock's current model catalog (`aws bedrock list-foundation-models`).
The Opus 4.7 ID uses the `global.anthropic.` prefix (e.g.,
`global.anthropic.claude-opus-4-7`). Confirm the Sonnet 4.6 equivalent
before the change lands — this is a hardcoded string that will fail at
runtime if wrong.

Override: `BEDROCK_MODEL_ID=global.anthropic.claude-opus-4-7` still works
for users who want Opus.

Anthropic path (`claude-sonnet-4-6`) is already the default since v5.3 — no
change there.

**Safety net:** `validate_ocr_plausibility` already cross-checks LLM amount
against `pdftotext` output and dates against email `internalDate ± 90
days`. If Sonnet drifts on a case Opus got right, the drift is flagged via
`_amountConfidence: "low"` or `_dateConfidence: "low"`, not silently
propagated. Opus remains the documented fallback.

### 6B — OCR prompt rules

Two new rules added to `scripts/core/prompts.py::get_ocr_prompt()`:

**Folio rule** (under `## 酒店水单(Guest Folio)专用字段`):

```
**⚠️ transactionDate 取值规则（酒店水单）：**
统一使用 **departureDate（退房日）** 作为 transactionDate。
理由：退房日是住宿服务完成的日期，与发票开票日对齐，是 P2 匹配依赖的字段。
不要取 arrivalDate、check-in、room charge date 等其他日期。
```

**Itinerary rule + new field** (under `## 网约车行程单专用字段`):

```
| applicationDate | string | 申请日期，从表头"申请日期：YYYY-MM-DD"提取（行程单专用） |

**⚠️ transactionDate 取值规则（行程单）：**
将 applicationDate 同时填入 transactionDate。
理由：申请日期是行程单的结算时间点，与对应发票的开票日期最接近。
不要取各笔行程的上车时间（不同行程日期不同）或行程起止范围的任一端点。
```

**Output example extension:** add one hotel-folio JSON sample and one
itinerary JSON sample to the prompt's `## 输出格式` section showing the
rule concretely.

**What the rule does NOT cover (explicit in prompt):**
- Hotel invoice (税务发票): `transactionDate` = 开票日. Unchanged.
- Meal / retail / train / phone invoices: `transactionDate` = 开票日. Unchanged.
- Ride-hailing invoice (not itinerary): `transactionDate` = 开票日. Unchanged.

### 6C — `applicationDate` downstream plumbing

The new field is read by the LLM into the OCR dict, serialized to cache,
and surfaced in the raw OCR JSON. It does **not** get a dedicated column
in CSV or report — it's provenance data. Consumers read `transactionDate`;
the prompt rule guarantees they're equal for itineraries.

No matching-logic change (`scripts/core/matching.py` reads `transactionDate`,
which is the v5.5 canonical field). No CSV/report column. No schema change.

back to reimbursement-helper for consistency — the OCR prompt is the
shared contract." CHANGELOG entry for v5.5 calls out this obligation so
the upstream sync isn't forgotten.
